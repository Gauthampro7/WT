Overview

SkillSwap is a full‑stack web application that lets college students list skills they can offer (e.g. tutoring, guitar, coding) or skills they are seeking, and request trades with other students. The project focuses on a polished user experience, secure authentication, and a clear separation between frontend and backend.

We built it as a single-page application (SPA) with React on the frontend and Supabase (PostgreSQL + Auth) on the backend, using SQL for schema, indexes, triggers, and Row Level Security (RLS).

Features

Core

  • Browse skills with real-time search and filters (category, type, campus)
  • Create, edit, and delete your own skills (Offering / Seeking)
  • Request trades with an optional message
  • Accept, decline, or mark trades as completed
  • Save (bookmark) skills and view them in Dashboard

UX & Design

  • 5 themes: Light, Dark, Midnight, Cyberpunk, Emerald
  • Glassmorphic UI with backdrop blur and soft borders
  • 3D hover tilt on skill cards (Framer Motion)
  • Confetti on successful trade request
  • Responsive layout and accessibility-friendly contrast

Auth & Security

  • Google OAuth via Supabase Auth
  • Row Level Security (RLS) on all tables
  • Users can only modify their own data; public read for active content

Dashboard

  • My Skills — manage your listings
  • Saved — bookmarked skills
  • Requests I sent — status (pending / accepted / declined / completed)
  • Requests I received — accept, decline, or mark completed
  • Notification badge for pending incoming requests

Frontend

Tech stack

React 18 (UI), Vite 7 (build & dev server), Tailwind CSS (utility-first styling with CSS variables for theming), Framer Motion (animations), Lucide React (icons), Canvas Confetti (celebration effect).

Structure

  • src/components/ — Reusable UI (SkillCard, SearchFilter, BentoGrid, ThemeSelector, LoginButton, modals, Dashboard tabs)
  • src/contexts/ — ThemeContext (theme state), AuthContext (user session from Supabase Auth)
  • src/services/ — skillsService, tradesService, savedSkillsService (all call Supabase client)
  • src/lib/supabase.js — Supabase client init with env URL and anon key

How we built the UI

We use a single App.jsx with two main views: Browse (search, filters, grid of SkillCards) and Dashboard (tabs: My Skills, Saved, Requests sent, Requests received). Modals handle: Request Trade (optional message), Trade success (confetti), Create Skill, and User Profile. Theming is done via CSS variables in index.css; the ThemeProvider toggles a class on the document root (e.g. theme-dark, theme-cyberpunk) so all components use the same palette.

Backend

Why Supabase

We chose Supabase so we could avoid writing a separate backend server. It provides PostgreSQL (database), Auth (including Google OAuth), and a JavaScript client that the frontend calls directly. All business rules and security are enforced in the database with Row Level Security (RLS) and triggers.

Authentication

Users sign in with Google OAuth via Supabase Auth. On first sign-in, a trigger creates a row in our public.users table (id, name, email, picture, university, location) so we can join skills and trade requests to profiles. The frontend stores the session and passes the JWT automatically on every Supabase request; RLS uses auth.uid() to restrict access.

API layer (no custom server)

We do not run an Express or Node server. The React app talks to Supabase using @supabase/supabase-js: we call supabase.from('skills').select(...), supabase.from('trade_requests').insert(...), etc. Filtering (category, type, search) is done via Supabase query builders; RLS ensures users only see and change what they are allowed to.

Database & SQL

Tables

TablePurpose
usersProfiles (id from auth.users, name, email, picture, university, location). Populated by trigger on signup.
skillsListings: title, description, category, type, location, status, user_id. Categories: Tech, Arts, Academic, Life Skills. Types: Offering, Seeking.
trade_requestsRequests: skill_id, requester_id, status (pending/accepted/declined/cancelled/completed), message. UNIQUE(skill_id, requester_id).
saved_skillsBookmarks: user_id, skill_id. UNIQUE(user_id, skill_id).

Schema (excerpt)

Skills table
CREATE TABLE public.skills (
  id UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  category TEXT NOT NULL CHECK (category IN ('Tech', 'Arts', 'Academic', 'Life Skills')),
  type TEXT NOT NULL CHECK (type IN ('Offering', 'Seeking')),
  location TEXT DEFAULT '',
  status TEXT DEFAULT 'active' CHECK (status IN ('active', 'completed', 'cancelled')),
  user_id UUID REFERENCES public.users(id) ON DELETE CASCADE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT now() NOT NULL
);

Indexes & full-text search

We added indexes on user_id, category, type, status, and created_at for faster queries. For search we use a GIN index on to_tsvector('english', title || ' ' || description) and filter with query.$text = { $search: searchTerm } in the client.

Row Level Security (RLS)

Every table has RLS enabled. Examples:

  • users: anyone can SELECT; only own row for UPDATE/INSERT.
  • skills: everyone can SELECT where status = 'active'; users can SELECT/UPDATE/DELETE their own; authenticated users can INSERT with their own user_id.
  • trade_requests: requesters see their requests; skill owners see requests on their skills; only authenticated users can INSERT (as requester); requester can UPDATE (e.g. cancel); skill owner can UPDATE (accept/decline/complete).
  • saved_skills: users see and manage only their own rows.

Triggers

handle_new_user: AFTER INSERT on auth.users, inserts a row into public.users with name, email, picture from OAuth metadata. handle_updated_at: BEFORE UPDATE on users, skills, trade_requests sets updated_at = now().

Architecture

Data flow (high level)

User (browser) → React + Supabase client → Supabase API (Postgres + Auth) → RLS policies → Tables

Trade request flow

Browse → Request Trade → Modal (optional message) → tradesService.create(skillId, message) → INSERT trade_requests → Success modal + confetti

Skill owner sees the request in Dashboard → Requests I received; they can Accept or Decline. Either side can later Mark as completed.

Conclusion

We built a complete, production-style skill exchange platform: frontend (React, Vite, Tailwind, Framer Motion), backend (Supabase: PostgreSQL + Auth), and SQL (schema, indexes, RLS, triggers). The app is secure (RLS, OAuth), responsive, and visually polished with multiple themes and animations. It is suitable as a college project demonstrating full-stack development, database design, and modern web tooling.
